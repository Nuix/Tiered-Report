Overview

This script allows you to define and generate reports on data ingested into one or more Nuix cases. The report is generated as Excel XLSX work book with one or more work sheets. For each work sheet you select one or more Item Aspects, which represent information which can be captured about an item such as:

  • Custodian
  • Item Kind
  • Tags
  • etc.

Each report sheet is then generated in a tiered manner.

For example if I were to specify a sheet in the report with the following aspects:

  • Evidence Name
  • Item Kind
  • Irregular Categories

I would get a report sheet which is first broken down by Evidence Name and then within that evidence further broken down by Item Kind and then finally broken down by Irregular Item Category.

For each tier the total number of items, total audited size and total file size is reported.

Settings Dialog

Main Tab

  • Report XLSX File: Where the report will be saved to.
  • Report Excluded Items as Well: When checked, excluded items will be included in reporting. When un-checked excluded items will NOT be included in reporting. Leaving this unchecked mimics the exclusion behavior of workbench.
  • Report Sizes using SI Units: When checked SI units will be used to calculate GB (1 GB = 1000^3 Bytes). When unchecked SI units will not be used (1GB = 1024^3 Bytes). Nuix workbench uses SI units when summing a size column so leave this option checked to get totals which match workbench. See here for more details on the topic.
  • Scope Query: A query which may be used to limit what items are reported on in each case. Leaving the value blank will report on all items in each case. When a query is provided, the script uses the hits of the provided query when collecting the report data.

Case Paths Tab

This tab allows you to either provide a list of case directories (one per case) or select use the currently open case if the script is being ran in the GUI and a case is open when the script is ran.

  • Allow Migration: Determines whether Nuix has permission to migrates cases when opening them. Only matters when using a list of case directories. If a case requires migration and this is unechecked, Nuix will throw a migration error for that case and the script will skip the given case.

Report Sheet Tabs

There are 15 report sheet tabs, each represents the settings for a possible report sheet to include in the final XLSX report work book. Each tab has the following settings:

  • Generate this Sheet: This sheet will only be generated in the final report if this is checked.
  • Sheet Name: The sheet name this sheet will be labelled with in the report. Note that Excel limits sheet names to 31 characters max.
  • Report Total Audited Count: Determines whether this report sheet will contain a column reporting the total audited item count for each tier.
  • Report Total Audited Size: Determines whether this report sheet will contain a column reporting the total audited size for each tier.
  • Report Total File Size: Determines whether this report sheet will contain a column reporting the total file size for each tier.
  • Report Total Digest Input Size: Determines whether this report sheet will contain a column reporting the total digest input size for each tier.
  • Report Total Corrupted Count: Determines whether this report sheet will contain a column reporting the total corrupted item count for each tier.
  • Report Total Deleted Count: Determines whether this report sheet will contain a column reporting the total deleted item count for each tier.
  • Sort Last Tier by Item Count Descending: When checked, the rows of the last tier will be sorted by how the item count in descending order. When unchecked, the rows of the last tier will be sorted alpha numerically by the last tier's values.
  • Include Category Header Rows: When checked the report will contain fewer redundant rows.
  • Use Spare Columns: When checked the report will populate fewer of the redundant column values.
  • Data Size Unit; Determines whether the selected size columns will be reported as Giga Bytes, Bytes or Dynamic. When Dynamic is chosen, sizes will be reported in the most appropriate unit (GB, MB, KB, Bytes).
  • Sub Scope Query: Determines a query that will scope the results of this report sheet.
  • Reported Item Aspects: This is where you determine what aspects of the items will be used to generate this report sheet. Note that the order of aspects matters so you may want to sort the rows of this table using the various included controls. See Item Aspects for more details about what each item aspect reports.

Important: Sub scope query determines a subset of the main scope query (found on the Main Tab). For example if the main scope query is kind:email and a given report sheet's sub scope query is kind:document the sheet will contain no results because there is no subset of items which are kind:email which are also kind:document.

Terms Tab

This is where you specify the terms which will be used by the aspect Terms (if it is being used in any report sheet).

Item Aspects

These represents aspects about an item which may be used to categorize items.

Tip: The order specified for aspects in a given report sheet is important to how the data is reported! Rows may be sorted in the settings dialog using the up and down arrow buttons or the "Sort Checked to Top" button at the far right. You will typically want to sort broader aspects (ex: Evidence Name) above more specific aspects (ex: Item Type).

Custom Metadata

You can configure the script to support reporting on custom metadata fields. You do this by editing the included CustomMetadataAspects.json file. The default example looks like this:

[
	{
		"enabled": false,
		"property_name": "UnconfiguredTemplate1",
		"report_label": "Unconfigured Template 1",
		"value_if_missing": "No Value"
	},
	{
		"enabled": false,
		"property_name": "UnconfiguredTemplate2",
		"report_label": "UnconfiguredTemplate2",
		"value_if_missing": "No Value"
	},
	{
		"enabled": false,
		"property_name": "UnconfiguredTemplate3",
		"report_label": "Unconfigured Template 3",
		"value_if_missing": "No Value"
	}
]
SettingDescription
enabledDetermines whether a given entry is available in the script. Provided mostly for an easy way to enable/disable entries. Set to true to enable.
field_nameThe name of the property metadata field in Nuix.
report_labelThe label used for the column header for this entry when generating the report. This allows you defined a report label which differs from the property metadata field name in Nuix.
value_if_missingThis allows you to define a fallback value to classify an item with when the item either does not have the property metadata field or has no value for the property metadata field.

How it Works

To start off I'm going to talk about how the script obtains data about items. Lets say we want to get a count of how many items have each tag in the case. I might start by getting a list of all the tags in a case and for each tag run a search for that tag. Imagine I have the following tags:

  • MyTag01
  • MyTag02

I could then run searches like:

  • tag:MyTag01
  • tag:MyTag02

For simpler reporting this works out well, but what if I want to further break down the reporting by what custodian is assigned to each tagged item in addition to counts for each tag. I would get a listing of all the custodians in the case and run searches for each. Imagine I have the following custodians:

  • Bob
  • John

I would then need to run searches not only for the tags, but each custodian and tag:

  • tag:MyTag01
  • tag:MyTag01 AND custodian:Bob
  • tag:MyTag01 AND custodian:John
  • tag:MyTag02
  • tag:MyTag02 AND custodian:Bob
  • tag:MyTag02 AND custodian:John

You can see as the amount of nested categories increses, the volume of searches begins to increase. What if you want to do this across many cases without having them all open at once? Then you would need to open each case and run all these searches in each case further increasing the number of searches.

Another approach may then be to record data about each item in a way which a case can be closed and new one opened while still allowing you to refer to the information you have collected from closed cases. So perhaps you decide instead to keep a record in memory of each aspect about an item you like to report on such as each tag it has, the custodian assigned, the item's type, etc. You can then mash this data together to report things like "What items are tagged MyTag01 and have a custodian of Bob". You could record that like this:

Item1:
	Size: 1.0 kb
	Tags: MyTag01, MyTag02
	Custodian: Bob
Item2:
	Size: 2.5 kb
	Tags: MyTag02
	Custodian: Bob
Item3:
	Size: 234 kb
	Tags: MyTag01
	Custodian: Bob
Item4:
	Size: 198 kb
	Tags: MyTag02
	Custodian: John
Etc..

Storing the data this way gets us closer, but we end up storing identical values over and over, which is a waste of memory and may not scale very well. So instead we might store it like this:

Has Tag 'MyTag01': Item1, Item3
Has Tag 'MyTag02': Item2, Item4
Has Custodian 'Bob': Item1, Item2, Item3
Has Custodian 'John': Item4

This way were storing each aspect once and tracking a list of what items have the given value for the given aspect.

Coming back to the nested categories, this approach can have problems scaling to large amounts of items. To determine how many items have tag MyTag01 and custodian Bob I need to look through all the items recorded for each and determine which items are present in both. Additionally the memory to track each item for a given aspect will be likely 64 bits (64 bit memory pointer) which can begin to add up on large data sets.

If I have 1 million items and only 1 is custodian "Bob" then thats just 64 bits (a drop in the bucket), but if all 1 million items have custodian "Bob" then I'm using at least 64 million bits to store this. While 64 million bits is still only about 8 MB, what if you have 100 tags, 100 custodians, 300 different types of items (500 aspect values) you could be looking at something more like 4 GB of memory (worst case). To allow reporting across multiple cases you have to think how big these numbers could actually begin to get as your total items reported gets into the millions.

Then there is still the consideration that determining what items have the tag MyTag01 and custodian Bob across 2 collections of potentially millions of entries begins to become a performance bottleneck.

Computers are great at bit manipulations and bits are a great way to store a large amount of data which will ever only be true or false (item has an aspect value or does not). So we turn to Bitmaps, not images like a BMP or PNG, but a string of bits:

0010

In the previous example we have a bitmap of 4 bits, with the 3rd bit set to true (1) and the rest of the bits set false (0). Now instead of referring to each item using 64 bits we can store whether 64 items have a given aspect value in 64 bits. To do this I use a library called RoaringBitmap. This library is also able to compress these bitmaps where there are long runs of only 0 or 1. Meaning that if I am tracking whether 1 million items have the tag MyTag01 and only 10 items actually have the tag, the resulting bitmap will be much smaller than 1 million bits. Even if all 1 million items have a tag, the bitmap will still be smaller than 1 milion bits due to compression of the Bitmap.

So now we have a way to store a true/false state for a large amount of items stating whether an item does or does not have a given value for a given aspect, what about nested category calculations? Computers are also very good at something called bit wise operations. We can leverage a bitwise AND to calculate category nesting.

So our previous example:

Has Tag 'MyTag01': Item1, Item3
Has Tag 'MyTag02': Item2, Item4
Has Custodian 'Bob': Item1, Item2, Item3
Has Custodian 'John': Item4

Now becomes:

Item1 = Bit 1 (1000)
Item2 = Bit 2 (0100)
Item3 = Bit 3 (0010)
Item4 = Bit 4 (0001)

With Tag 'MyTag01': 1010
With Tag 'MyTag02': 0101
With Custodian 'Bob': 1110
With Custodian 'John': 0001

If I want to know what is tagged MyTag01 and has custodian Bob we just AND those two bitmaps together:

Bitwise AND cheat sheet:

Bit ABit BAND value
000 (false)
100 (false)
111 (true)
010 (false)
  With Tag 'MyTag01': 1010
With Custodian 'Bob': 1110
                      ====
               ANDed: 1010

Result: Item1, Item3

Over a large amount of items/bits, thanks to RoaringBitmaps, this operation performs pretty well.
While using this approach can add some complexity I get efficient storage of which items have what aspect values at scale, a fast way to determine what items have a given combination of aspect values and I only need to collect the information about each aspect value of each item once while being able to re-combine it in different ways to generate different reports.

So now that some of the technical aspects are out of the way, what's the scripts workflow?

  1. You provide a scope query
  2. You choose the cases to report on
  3. For each sheet you would like reported, you choose one or more aspects and the order to report them in (order is important!)
  4. The script opens each case:
    1. The script runs your scope query in the case
    2. For each resulting item, each aspect collects its data about the item (ie: "Tags" aspect records what tags an item has)
  5. Step 4 is repeated for each case
  6. Each sheet is then generated by ANDing each tier's bitmap to the previous tier's ANDed bitmap
